# cpp-logger-library

Многопоточная библиотека для асинхронной записи сообщений в журнал (текстовый файл) с уровнями важности. Реализована на C++17 с использованием только STL.

---

## 📋 Описание

Проект состоит из двух частей:

1.  **Динамическая библиотека (`liblogger.so`)** — предоставляет потокобезопасный интерфейс для записи сообщений с тремя уровнями важности (`INFO`, `WARNING`, `ERROR`). Поддерживает фильтрацию по уровню и изменение уровня по умолчанию во время работы.
2.  **Консольное приложение (`logger_app`)** — многопоточная обёртка над библиотекой. Принимает от пользователя сообщения и уровень важности, после чего асинхронно передаёт их в отдельный поток для записи в файл, не блокируя ввод.

## 🛠 Требования

- **Операционная система:** Ubuntu / Debian (актуальные версии)
- **Компилятор:** GCC с поддержкой C++17
- **Сборщик:** `make` (GNU Make)
- **Зависимости:** только стандартная библиотека C++ (STL)

## 🚀 Сборка проекта

Сборка управляется через `Makefile`. В корневой директории проекта выполните:

```bash
make all
```

Эта команда соберёт обе цели:
- `liblogger.so` — динамическую библиотеку.
- `logger_app` — тестовое консольное приложение.

### Доступные цели Makefile

| Команда          | Действие                                                                 |
|------------------|--------------------------------------------------------------------------|
| `make lib`       | Собрать только динамическую библиотеку (`liblogger.so`).                 |
| `make app`       | Собрать только консольное приложение (`logger_app`).                     |
| `make clean`     | Удалить все объектные файлы, исполняемые файлы и саму библиотеку.        |
| `make all`       | Выполнить сборку библиотеки и приложения (действие по умолчанию).        |

## 📦 Использование

### 1. Консольное приложение (`logger_app`)

Приложение запускается с двумя обязательными параметрами:

```bash
./logger_app <имя_файла_журнала> <уровень_по_умолчанию>
```

**Пример:**
```bash
./logger_app app.log INFO
```

**Уровни важности (регистронезависимы):**
- `INFO`
- `WARNING`
- `ERROR`

#### Режим работы

После запуска приложение выводит приглашение `> ` и ожидает ввод от пользователя.

**Формат ввода:**
```
<сообщение> [<уровень>]
```

- Если указан только текст — сообщение записывается с уровнем по умолчанию (заданным при запуске).
- Если указан уровень — используется именно он (если уровень валидный).

**Примеры:**
```
> Запуск системы
> Ошибка подключения ERROR
> Предупреждение о перегреве WARNING
```

#### Управление во время работы

- Введите `change_level <НОВЫЙ_УРОВЕНЬ>`, чтобы изменить уровень по умолчанию.  
  Пример: `change_level ERROR` — теперь все сообщения без указания уровня будут записываться как `ERROR`.
- Введите `exit` или нажмите `Ctrl+C` для завершения работы.

Приложение корректно обрабатывает ошибки файловой системы (например, недоступность файла) и уведомляет пользователя через консоль.

---

### 2. Библиотека (`liblogger.so`)

Библиотека предоставляет простой класс `Logger` с публичным интерфейсом:

#### Конструктор
```cpp
Logger(const std::string& filename, Level defaultLevel);
```
- `filename` — путь к файлу журнала.
- `defaultLevel` — начальный уровень фильтрации.

#### Методы
```cpp
void log(const std::string& message, Level level);
```
Записывает сообщение в файл, если `level >= defaultLevel`. Время сообщения проставляется автоматически в формате `[YYYY-MM-DD HH:MM:SS]`.

```cpp
void setDefaultLevel(Level level);
```
Изменяет уровень фильтрации во время работы.

```cpp
Level getDefaultLevel() const;
```
Возвращает текущий уровень по умолчанию.

## ⚙️ Внутреннее устройство и подходы

- **Многопоточность:** Для асинхронной записи используется отдельный поток-обработчик, взаимодействие с которым осуществляется через потокобезопасную очередь на основе `std::mutex` и `std::condition_variable`.
- **Фильтрация:** Уровни важности реализованы через `enum class Level`, что обеспечивает строгую типизацию. Сравнение происходит как численное (`int`), где `ERROR` > `WARNING` > `INFO`.
- **Обработка ошибок:** Все операции с файлами обёрнуты в блоки `try-catch` с автоматическими повторными попытками (до 3 раз). Ошибки выводятся в `std::cerr`, но не приводят к аварийному завершению программы.
- **ООП:** Код спроектирован вокруг класса `Logger`, инкапсулирующего состояние (уровень, файловый поток, поток-обработчик). Логика консольного интерфейса вынесена в отдельный модуль `console_handler`.

## 🧪 Проверка работоспособности

Рекомендуемый сценарий тестирования:

1.  Соберите проект: `make all`
2.  Запустите приложение: `./build/logger_app test.log INFO`
3.  Введите несколько сообщений:
    ```
    > Первое сообщение
    > Ошибка соединения ERROR
    > Предупреждение WARNING
    ```
4.  Измените уровень по умолчанию:
    ```
    > change_level ERROR
    ```
5.  Введите сообщение без уровня:
    ```
    > Это сообщение будет проигнорировано
    ```
6.  Проверьте содержимое файла `test.log` — должны быть только сообщения с уровнем `ERROR` и `WARNING` (если `WARNING` >= `ERROR` — нет, но для наглядности уровни заданы так, что `ERROR` выше, поэтому `WARNING` не попадёт).
7.  Завершите приложение: `exit`
